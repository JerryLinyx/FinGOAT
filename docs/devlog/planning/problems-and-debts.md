# Problems And Debts

本文档按重要程度整理当前系统的核心问题、定位过程、根因、方案比较和推荐路线。

## P0-1 Python 任务状态仅存在内存中

### 问题描述

`langchain-v1/trading_service.py` 当前使用进程内字典 `analysis_tasks` 保存任务状态。

### 影响范围 / 严重性

- 服务重启会丢失运行期任务状态
- 多实例部署不可用
- 无法可靠扩展 worker 数量
- Go 查询结果时强依赖 Python 当前进程的存活状态

### 如何定位

- 直接检查 Python 服务任务实现
- 对照 Go 查询任务结果的逻辑
- 确认当前状态并未写入 Redis 或 PostgreSQL

### 排查和验证过的假设

- 假设 1：任务状态可能已持久化到 DB  
  结果：排除。DB 只保存了任务和决策结果，不保存完整运行态。

- 假设 2：任务状态可能在 Redis 中有副本  
  结果：排除。Redis 当前主要用于文章缓存和点赞。

- 假设 3：Python 仅临时缓存，但 Go 才是唯一真相源  
  结果：排除。Go 在任务进行中仍需回查 Python 内存状态。

### 最终根因

当前实现是 PoC 级任务模型：Python 同时承担执行器和运行态状态拥有者，且运行态保存在单进程内存里。

### 如何解决

推荐方案：

- PostgreSQL 保存业务真相与最终结果
- Redis 保存运行态、协调态和队列态
- Python 只负责执行
- Go 作为唯一业务 API 入口

### 方法论 / 技术路径

- 单一真相源原则
- 持久态与运行态分层
- 快状态与慢状态分层

### 其他方案

#### 方案 A

继续由 Python 持有任务状态，只把它做得更耐久

- 优点：改动相对少
- 缺点：业务状态仍然从 Go 分叉出去，边界不清

#### 方案 B

只用 PostgreSQL 管所有状态

- 优点：概念简单
- 缺点：不适合高频运行态协调和队列化

#### 方案 C

PostgreSQL 管持久业务状态，Redis 管运行时协调

- 优点：职责匹配数据生命周期
- 缺点：实现更复杂，需要定义清晰边界

### 最终选择原因

选择方案 C，因为它最符合当前系统已经具备的 Go/PostgreSQL/Redis 结构，也最适合 MVP 阶段的可靠性要求。

### 当前状态

## P0-2 系统状态源不唯一

### 问题描述

Go 保存任务和决策，Python 又单独保存运行中的任务状态，系统不存在明确的唯一状态源。

### 影响范围 / 严重性

- 状态可能漂移
- 查询逻辑复杂
- 责任边界不清
- 后续扩容和恢复流程难设计

### 如何定位

- 分析 `backend/controllers/trading_controller.go`
- 对照 Python `analysis_tasks`
- 追踪“提交任务 -> 轮询结果 -> 落库”的完整路径

### 排查和验证过的假设

- 假设 1：Go 已完整拥有任务生命周期  
  结果：排除，运行中状态仍要去 Python 查。

- 假设 2：Python 只是执行器，不保存状态  
  结果：排除，Python 当前实际保有运行态。

### 最终根因

主线架构方向正确，但服务边界尚未完全收紧，导致 Go 和 Python 在任务状态层面“双重拥有”。

### 如何解决

把 Go 定义为唯一业务入口与状态提供方，把 Python 限定为执行器。

### 方法论 / 技术路径

- 服务边界收紧
- 单一来源原则
- 业务 API 与执行引擎分层

### 其他方案

#### 方案 A

让 Python 成为主状态服务

- 优点：最贴近当前执行过程
- 缺点：削弱 Go 网关角色，业务边界更混乱

#### 方案 B

让 Go/PostgreSQL 成为业务真相源，Redis 负责运行时状态

- 优点：更符合当前产品外壳 + 执行引擎分层
- 缺点：需要重构部分读写逻辑

### 最终选择原因

选择方案 B，因为它延续现有主架构，不需要推翻语言和服务分工。

### 当前状态

待解决。

## P0-3 Go/Python 契约弱类型

### 问题描述

Go 当前使用 `map[string]interface{}` 接收 Python 响应，并对字段做强制类型断言。

### 影响范围 / 严重性

- 上游格式波动可触发运行时错误
- 内部接口演进成本高
- 前端结果结构难以长期稳定

### 如何定位

- 检查 Go Trading controller 的 Python 响应解析逻辑
- 发现 `action`、`confidence` 等字段直接做类型断言

### 排查和验证过的假设

- 假设 1：问题只在前端解析层  
  结果：排除。风险点在 Go 内部。

- 假设 2：弱类型只是临时写法，不影响稳定性  
  结果：排除。只要 Python payload 漂移，Go 就会出风险。

### 最终根因

PoC 阶段优先打通链路，内部 API 契约没有被当作正式接口维护。

### 如何解决

- 定义稳定的 JSON schema
- Python 明确输出字段
- Go 使用强类型结构体解析
- 统一错误结构和状态字段

### 方法论 / 技术路径

- 先收紧 schema，再讨论协议
- 内部 API 也按正式接口对待

### 其他方案

#### 方案 A

继续 REST，但做强类型 schema

- 优点：成本低，直接解决当前问题
- 缺点：协议层能力不如 gRPC

#### 方案 B

立即迁移到 gRPC

- 优点：IDL 强约束、代码生成、后续 streaming 更方便
- 缺点：迁移成本高，且不能直接解决任务状态双源问题

### 最终选择原因

选择方案 A。当前根因是 schema，而不是协议。

### 当前状态

待解决。

## P1-1 Redis 职责过轻

### 问题描述

Redis 当前只承担文章缓存和点赞计数，没有进入核心分析任务主链路。

### 影响范围 / 严重性

- 任务缺少协调层
- 运行态状态无合适承载位置
- 外部数据抓取缺少共享缓存和锁

### 如何定位

- 检查 Go 代码中 Redis 的使用点
- 对照 Python 和 TradingAgents 的执行需求

### 排查和验证过的假设

- 假设 1：Redis 已承担任务缓存  
  结果：排除。

- 假设 2：当前设计不需要 Redis 进入核心链路  
  结果：初步排除。对于异步任务系统，Redis 明显有更大价值。

### 最终根因

v0.1.0 把 Redis 当成辅助缓存使用，没有把它视为任务协调基础设施。

### 如何解决

扩展 Redis 职责：

- 任务队列
- 任务运行态缓存
- 分布式锁
- vendor 抓取去重
- 热结果缓存

### 方法论 / 技术路径

- 按数据生命周期分层
- 按一致性要求区分 PostgreSQL 与 Redis 角色

### 其他方案

#### 方案 A

继续只把 Redis 当缓存

- 优点：实现简单
- 缺点：浪费现有基础设施，无法支持更稳的任务系统

#### 方案 B

把任务协调能力纳入 Redis

- 优点：更适合短生命周期运行态和高频写入
- 缺点：需要补生命周期设计

### 最终选择原因

选择方案 B。Redis 非常适合当前异步任务系统的协调层角色。

### 当前状态

待解决。

## P1-2 分析透明度不足

### 问题描述

主线前端主要显示最终结果，用户和开发者难以知道 agent 执行到哪里、耗时多少、哪些中间结果已产生。

### 影响范围 / 严重性

- 调试成本高
- 用户信任感不足
- 无法准确发现慢点和失败阶段

### 如何定位

- 对比主线前端和 `origin/dev_gq2142`
- 观察主线 TradingAnalysis 的展示范围

### 排查和验证过的假设

- 假设 1：当前后端已具备足够的阶段数据，只是前端没展示  
  结果：部分成立。主线没有系统化阶段输出；分支里有更成熟方案。

- 假设 2：只要做日志就够了  
  结果：初步排除。日志对用户价值弱，对产品层可解释性不足。

### 最终根因

v0.1.0 优先完成闭环，尚未把“过程可解释性”纳入主线。

### 如何解决

- 吸收 `origin/dev_gq2142` 中的阶段耗时和关键输出设计
- 在主线 schema 中显式加入 `stage_times` 和 `key_outputs`
- 前端增加阶段式视图

### 方法论 / 技术路径

- 先观测，再优化
- 统一阶段模型，再做 UI

### 其他方案

#### 方案 A

只做后端日志

- 优点：实现简单
- 缺点：产品价值低

#### 方案 B

后端输出结构化阶段信息，前端同步展示

- 优点：对用户和开发者都更有价值
- 缺点：要先稳定 schema

### 最终选择原因

选择方案 B，因为它同时提升调试效率和产品可解释性。

### 当前状态

部分解决，能力已在分支中出现，但未进入主线。

## P1-3 配置体系分散

### 问题描述

当前配置分散在 Go YAML、环境变量、Python 默认配置、Docker Compose 中，且部分代码存在硬编码覆盖。

### 影响范围 / 严重性

- 调试时难以判断实际生效配置
- 部署行为不透明
- 跨环境迁移成本高

### 如何定位

- 检查 Go config 和 DB 初始化逻辑
- 检查 Python 默认配置
- 对照 Docker Compose 环境变量

### 排查和验证过的假设

- 假设 1：配置来源很多，但优先级已经清晰  
  结果：排除。

- 假设 2：只有文档混乱，运行逻辑清晰  
  结果：排除。部分值在运行时被硬编码。

### 最终根因

多语言、多服务系统在快速开发中逐步叠加配置方式，但没有统一规则。

### 如何解决

- 规定 source of truth
- 明确优先级：默认值 < 文件配置 < 环境变量
- 消除不必要的硬编码覆盖

### 方法论 / 技术路径

- 配置层次明确化
- 配置行为可追踪

### 其他方案

#### 方案 A

全量切 env-only

- 优点：部署统一
- 缺点：本地开发可读性较差

#### 方案 B

保留文件配置，允许 env override

- 优点：兼顾开发和部署
- 缺点：需要严格规范优先级

### 最终选择原因

选择方案 B，更适合当前项目阶段。

### 当前状态

待解决。
