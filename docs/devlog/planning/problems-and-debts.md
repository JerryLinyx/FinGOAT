# Problems And Debts

本文档按优先级记录“当前真实状态”：哪些问题已落地修复，哪些仍是 v0.2 的核心技术债。

## P0-1 任务运行态在 Python 内存（历史问题）

### 问题描述

历史上任务状态仅在 Python 进程内维护，重启后会丢失。

### 影响范围 / 严重性

- 重启丢状态
- 多实例不可用
- Go 查询依赖单进程存活

### 如何定位

- 检查 `langchain-v1/trading_service.py` 的任务状态实现。
- 对照 Go 查询路径与 DB 落库行为。

### 排查和验证过的假设

- 假设：状态只在 DB。  
  结果：排除，运行态未完整入库。
- 假设：Redis 仅做缓存。  
  结果：历史阶段成立，后续已调整。

### 最终根因

PoC 阶段把执行器和运行态所有权放在同一 Python 进程内。

### 如何解决

- Go 负责创建任务并落库。
- Redis 承担队列与 runtime state。
- Python 作为 worker 从 Redis 消费并回写 runtime。

### 方法论 / 技术路径

- 持久态和运行态分层
- 单一业务入口
- 快/慢状态分治

### 其他方案

- 方案 A：仅用 PostgreSQL 管理运行态（实现简单，但高频协调不合适）
- 方案 B：仅用 Redis（持久业务记录不够稳）
- 方案 C：PostgreSQL + Redis 分工（已采用）

### 最终选择原因

方案 C 同时满足持久性和运行态协调效率，且与现有栈兼容。

### 当前状态

已解决（主链路已落地）。

## P0-2 状态源不唯一（由“未治理”转为“部分治理”）

### 问题描述

历史上 Go 与 Python 对任务状态“双重拥有”，边界不清。

### 影响范围 / 严重性

- 状态漂移风险
- 对账逻辑复杂

### 如何定位

- 追踪提交、轮询、终态回写路径。
- 检查 `backend/controllers/trading_runtime.go` 的 reconciliation。

### 排查和验证过的假设

- 假设：Go 已完全单一真相源。  
  结果：部分成立，运行态仍在 Redis，需对账回写。

### 最终根因

运行态与持久态天然分层，但对账策略在早期未明确。

### 如何解决

- PostgreSQL 作为持久业务真相。
- Redis 作为运行态真相。
- Go 查询路径触发对账并回写终态。

### 方法论 / 技术路径

- 业务真相源明确化
- 运行态 -> 持久态收敛

### 其他方案

- 方案 A：让 Python 成为对外状态服务（边界更乱）
- 方案 B：保留 Go 业务入口 + Redis 运行态（已采用）

### 最终选择原因

延续现有架构分层，最小代价获得稳定性。

### 当前状态

部分解决：核心链路已稳定，仍可补“后台定时对账/修复”能力。

## P0-3 Go/Python 契约弱类型（当前核心未决问题）

### 问题描述

关键返回对象仍包含较多动态结构（如 `analysis_report`、`reasoning`），强约束不足。

### 影响范围 / 严重性

- schema 漂移引发运行时风险
- 前后端演进成本高

### 如何定位

- 检查 Go 的 runtime/response 结构定义。
- 对照 Python 写入 runtime 的 JSON 形态。

### 排查和验证过的假设

- 假设：仅前端层有弱类型风险。  
  结果：排除，Go 内部聚合层同样受影响。

### 最终根因

PoC 阶段优先打通流程，未把内部契约当正式接口治理。

### 如何解决

- 定义稳定 JSON schema（含错误模型）。
- Go 侧收紧结构体和校验。
- Python 侧固定关键字段与版本化约束。

### 方法论 / 技术路径

- 先 schema，后协议
- 契约测试先行

### 其他方案

- 方案 A：REST + 强 schema（推荐）
- 方案 B：直接迁移 gRPC（约束强，但迁移成本更高）

### 最终选择原因

当前痛点在契约治理，不在协议本身。

### 当前状态

待解决（P0）。

### 问题描述

### 影响范围 / 严重性

### 如何定位

- 根据 `task-backlog.md` 与对应 records 的验证记录追踪。
- 对照当前代码中的 provider/embedding 路由改动。

### 排查和验证过的假设

- 假设：provider 问题已全部关闭。  
  结果：排除，仍保留待验证项。

### 最终根因

多 provider + 多 vendor 路由在快速迭代中容易出现隐式 fallback。

### 如何解决

- 建立 provider 回归矩阵（model、embedding、news/vendor 路由）。
- 将关键路径验证自动化并固定在发布前检查清单中。

### 方法论 / 技术路径

- 关键日志与断言双校验

### 其他方案

- 方案 B：回归矩阵自动化（已选）

### 最终选择原因

### 当前状态

部分解决（仍有待关闭验证项）。

## P0-5 Go/FastAPI 任务 API 边界重叠

### 问题描述

当前主业务路径已经由 Go 统一承接，但 Python `trading_service` 仍暴露 `analyze/query` 任务接口，形成对外能力重叠。

### 影响范围 / 严重性

- 可能绕过 Go 网关的鉴权、任务语义和持久化约束。
- 团队协作中易出现双入口混用，增加排障和回归复杂度。

### 如何定位

- 检查 Go 路由与控制器调用路径（主流程已是 Go + Redis）。
- 检查 Python FastAPI 路由，确认任务 API 仍对外可调用。

### 排查和验证过的假设

- 假设：Go 仍通过 HTTP 调 Python `analyze` 完成主流程。  
  结果：排除。当前主流程为 Go 入队 + Python worker 消费。
- 假设：Python 任务 API 已经下线。  
  结果：排除。接口仍存在。

### 最终根因

架构已演进为“Go 外部边界 + Python worker”，但接口收口尚未完成，遗留了历史兼容路径。

### 如何解决

- Go 作为唯一外部交易 API 边界。
- Python 收敛为内部执行服务，生产路径限制/下线任务对外接口。
- 增加边界回归检查，防止重新引入双入口。

### 方法论 / 技术路径

- 单边界原则（Single External API Boundary）
- 边界先于约定

### 其他方案

- 方案 A：保留双入口并通过文档约定只用 Go
  - 优点：改动小
  - 缺点：长期不可控，容易回退
- 方案 B：收敛为 Go 单入口（推荐）
  - 优点：职责清晰，契约与安全统一
  - 缺点：需要迁移旧脚本/调试习惯

### 最终选择原因

方案 B 更符合 v0.2.0 的治理目标，能一次性清除接口层重复责任。

### 当前状态

待解决（P0）。

## P1-6 用户域能力缺口（账号模型/登录方式/自助配置）

### 问题描述

当前用户域仍是最小 PoC 形态，缺少 v0.2 所需的账号与配置能力：

- 用户表模型仍偏简化（`username + password`）
- 未支持 email 登录
- 未提供 Profile 与 API Key 自助配置页面

### 影响范围 / 严重性

- 限制账号体系扩展（身份标准化、找回、风控策略）
- 限制用户自助能力（资料维护、provider 密钥管理）
- 影响后续多 provider 策略的产品化落地

### 如何定位

- 检查 Go 认证路由与用户模型定义。
- 对照前端页面结构，确认缺失 Profile/API Key 页面。
- 对照 v0.2 backlog 与新增需求记录。

### 排查和验证过的假设

- 假设：现有用户名登录足够支撑 v0.2。  
  结果：排除。账号扩展与用户自助配置需求已明确存在。
- 假设：API Key 配置可继续仅靠运维环境变量。  
  结果：部分成立，仅适合早期内部使用，不利于产品化。

### 最终根因

v0.1 阶段优先保证交易链路闭环，用户域只实现了最小认证能力，未同步建设账号与配置体系。

### 如何解决

- 重设计用户表：email 唯一约束、password hash 语义、profile 字段预留。
- 增加 email 注册/登录与历史用户名兼容迁移策略。
- 新增 Profile 页面及对应读写 API。
- 新增用户 API Key 配置页面与后端安全存储契约（密钥不明文回显）。

### 方法论 / 技术路径

- 先数据模型，再接口契约，再页面接入
- 先兼容迁移，再逐步收口旧路径
- 敏感信息最小暴露原则

### 其他方案

- 方案 A：维持现状，仅补前端页面
  - 优点：实现快
  - 缺点：底层模型仍不满足后续扩展
- 方案 B：先做用户域基础重构，再补页面（推荐）
  - 优点：一次性打通模型、接口、安全边界
  - 缺点：短期改动范围更大

### 最终选择原因

方案 B 更符合 v0.2 从“跑通”到“可持续 MVP”的目标，避免未来反复返工。

### 当前状态

待解决（P1，账号模型与 email 登录可按 P0 子项先行）。

## P1-8 Trading 分析重复运行防护缺失

### 问题描述

Dashboard 的 TradingAgents 分析提交流程缺少重复运行防护。  
当用户提交“同代码 + 同日期”分析时，系统不会提醒是否已分析过。

### 影响范围 / 严重性

- 产生不必要的重复任务执行成本
- 任务列表噪音增加，影响历史分析可读性
- 用户可能误触发重复分析而不自知

### 如何定位

- 审查当前提交交互，未发现命中历史记录后的确认弹窗。
- 对照已有历史分析能力，确认可作为判重信息源。

### 排查和验证过的假设

- 假设：允许重复提交不需要额外交互。  
  结果：排除。目标是减少“无意重复”，不是禁止“有意重跑”。
- 假设：仅前端本地历史即可完全判重。  
  结果：排除。前端数据可能不完整，存在漏判。

### 最终根因

v0.1.2 的重点在任务链路稳定与阶段可视化，提交前的“重复运行成本控制”未纳入交互闭环。

### 如何解决

- 定义判重口径：同用户 + 同代码 + 同分析日期。
- 命中后弹窗二次确认，用户确认才继续提交任务。
- 后端提供可复用的判重查询语义，前端仅负责交互确认。

### 方法论 / 技术路径

- 成本防护优先于硬阻断
- 交互确认与后端语义判重组合

### 其他方案

- 方案 A：完全禁止重复提交
  - 优点：最强去重
  - 缺点：剥夺用户主动重跑场景
- 方案 B：仅前端提醒
  - 优点：开发快
  - 缺点：可靠性受前端数据完整度影响
- 方案 C：后端判重 + 前端确认（推荐）
  - 优点：兼顾可靠性与灵活性
  - 缺点：需要接口协同改造

### 最终选择原因
